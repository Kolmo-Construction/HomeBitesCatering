createInsertSchema(formPages) will expect JavaScript object keys matching the Drizzle table definition:
formId: number (this is added by the server route handler before parsing)
pageTitle: string | null | undefined (from text("page_title"))
pageOrder: number (from integer("page_order").notNull()). Since it's notNull and has no DB default, drizzle-zod will make this a required number in the Zod schema.
description: string | null | undefined (from text("description"))
Discrepancy Identified (Root Cause):

The pageOrder field is the issue.

Server-side expectation: insertFormPageSchema (derived from formPages where pageOrder is NOT NULL and has no default) will require pageOrder to be a number.
Client-side formPageSchema (FormEditor.tsx): pageOrder: z.number().optional().
Client-side PageFormDialog (FormEditor.tsx): The defaultValues for the dialog (line 282) only sets pageTitle and description. It does not set or provide an input for pageOrder.
TypeScript

defaultValues: initialData ? { ... } : {
  pageTitle: "",
  description: "",
  // pageOrder is missing here for new pages
}
Client-side createPageMutation (FormEditor.tsx): The pageData passed to JSON.stringify(pageData) (line 378) will therefore likely be missing the pageOrder field when creating a new page.
When the server receives the req.body for creating a new page, it will look like:

JSON

{
  "pageTitle": "Basic Information",
  "description": "basic info"
  // pageOrder is missing
}
The server-side pageSchema.safeParse({ ...req.body, formId }) will then fail because the Zod schema generated from insertFormPageSchema requires pageOrder to be present and a number.

Solution:

The client needs to ensure pageOrder is included in the data sent to the server when creating a new page.

1. Calculate pageOrder on the client before creating a page:
When handleSavePage is called for a new page (i.e., !editPageData), you need to determine the next available pageOrder.

Modify handleSavePage in clea/client/src/pages/FormEditor.tsx:

TypeScript

// clea/client/src/pages/FormEditor.tsx (lines 402-407)
const handleSavePage = async (dataFromDialog) => { // Renamed pageData to dataFromDialog for clarity
  if (editPageData) {
    await updatePageMutation.mutateAsync({
      pageId: editPageData.id,
      pageData: dataFromDialog, // For updates, pageOrder should already exist or be part of dataFromDialog if editable
    });
  } else {
    // For new pages, calculate the next pageOrder
    const existingPages = pagesData?.data || []; // pagesData is from the useQuery hook
    const nextOrder = existingPages.length > 0
      ? Math.max(...existingPages.map(p => p.pageOrder || p.page_order || 0)) + 1 // Handle potential snake_case from server
      : 1; // Start with 1 if no pages exist

    const pageDataToSend = {
      ...dataFromDialog,
      pageOrder: nextOrder // Add pageOrder here
    };

    const newPage = await createPageMutation.mutateAsync(pageDataToSend);
    if (newPage) {
      setSelectedPage(newPage); // Select the newly created page
    }
  }
};
2. Ensure the server expects camelCase keys for the body (consistent with previous fix):
The JSON.stringify(pageDataToSend) in createPageMutation (line 378) will use the keys from pageDataToSend.

pageTitle (from dataFromDialog) is already camelCase.
description (from dataFromDialog) is already camelCase.
pageOrder (added above) is camelCase.
The server's insertFormPageSchema generated by drizzle-zod from formPages (which uses camelCase JS keys like pageTitle, pageOrder) will correctly expect these camelCase keys. Drizzle ORM will then map these to the snake_case database columns (page_title, page_order).
