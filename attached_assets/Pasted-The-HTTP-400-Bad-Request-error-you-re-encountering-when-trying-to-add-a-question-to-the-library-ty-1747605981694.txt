The HTTP 400 (Bad Request) error you're encountering when trying to add a question to the library typically indicates that the data sent from the client to the server does not match what the server expects, often due to a validation failure.

Let's break down the probable causes and solutions by examining the relevant code sections:

1. Backend Expectations (server/questionLibraryRoutes.ts and shared/form-schema.ts):

The backend route POST /api/form-builder/library-questions in server/questionLibraryRoutes.ts uses a Zod schema (locally defined as questionLibrarySchema) to validate the request body (req.body). This schema expects the following structure:

TypeScript

// Simplified from server/questionLibraryRoutes.ts
const questionLibrarySchema = z.object({
  libraryQuestionKey: z.string().min(1).max(255),
  defaultText: z.string().min(1),
  questionType: z.enum([/* various question types */]),
  defaultMetadata: z.any().optional(), // For general metadata
  defaultOptions: z.any().optional(), // Specifically for choice-based questions (checkbox, radio, dropdown)
  category: z.string().optional()
});
Additionally, the Drizzle database schema for the question_library table (defined in shared/form-schema.ts) has distinct columns for defaultMetadata and defaultOptions.

The route handler also explicitly looks for req.body.matrixRows and req.body.matrixColumns if questionType is 'matrix'.

3. Identifying the Mismatches:

Extraneous Top-Level Fields:
The client sends default_required, helper_text, and placeholder as top-level properties in the submission object.
The backend's Zod schema (questionLibrarySchema in server/questionLibraryRoutes.ts) does not define these fields at the top level. This is a primary cause for Zod validation failure, leading to the 400 error. These properties, if intended to be stored, should generally be part of the defaultMetadata JSONB field.
Incorrect Placement of defaultOptions:
The backend schema and Drizzle table structure expect defaultOptions as a distinct top-level field for choice-based questions (like dropdowns, radio buttons, checkboxes).
The client's prepareMetadataForSubmission function currently includes the options array (for choice questions) inside the object that becomes default_metadata (as metadata.options ). The main submission object then only sends default_metadata, not a separate default_options. This is a clear mismatch.
