The cloning of matrix rows and columns should be handled independently. Rows should be cloned if they exist, and columns should be cloned if they exist, regardless of the presence of the other.

Modify the matrix cloning logic in server/questionLibraryRoutes.ts within the POST /api/form-builder/library-questions/:id/clone route as follows:

const isMatrixType = originalQuestion.question_type === 'matrix' || originalQuestion.questionType === 'matrix';

if (isMatrixType) {
  // Clone rows if they exist
  if (originalQuestion.matrixRows && originalQuestion.matrixRows.length > 0) {
    console.log(`Cloning ${originalQuestion.matrixRows.length} matrix rows for new question ID: ${newQuestion.id}`);
    for (const row of originalQuestion.matrixRows) {
      const createRowSql = `
        INSERT INTO library_matrix_rows

      // Ensure row properties are accessed correctly (snake_case from DB)

        newQuestion.id,

      `;
      // Ensure column properties are accessed correctly (snake_case from DB)
      const columnValues = [
        newQuestion.id,
        column.column_key,
        column.header,
        column.cell_input_type,
        column.default_metadata ? JSON.stringify(column.default_metadata) : null, // Ensure metadata is stringified

      ];

Explanation of Changes:

The condition to enter the matrix cloning block now correctly checks if the originalQuestion is of type 'matrix'.
Inside this block, cloning for matrixRows and matrixColumns is done in separate if blocks. This ensures that rows are cloned if present, and columns are cloned if present, independently of each other.
Added JSON.stringify for default_metadata when inserting cloned rows and columns, as the database expects JSONB types to be stringified when inserted via raw SQL queries if they are passed as JavaScript objects. The getQuestionWithDetails function retrieves them as parsed objects.
Route Precedence:

It's important to note that server/index.ts registers routes from questionLibraryRoutes.ts before specifically registering fixedCloneRoute.ts for the same POST /api/form-builder/library-questions/:id/clone path. In Express, the first route handler defined for a path and method takes precedence. Therefore, the problematic cloning logic resides in server/questionLibraryRoutes.ts, and the fix should be applied there. The code in fixedCloneRoute.ts is not currently active unless the registration order in server/index.ts is changed.
