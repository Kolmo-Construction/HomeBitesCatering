Phase 2: Multi-Step Inquiry Form (Core Data Capture)

General Behavior for All Steps:

Step Indication: Clearly display the current step number and total number of relevant steps (e.g., "Step 2 of 8"). A visual progress bar is recommended.
Navigation:
"Next" button: Validates the current step's required fields. If valid, saves current step's data to the main form state and proceeds to the next relevant step. If invalid, displays clear, field-specific error messages.
"Previous" button: Saves current step's data to the main form state and navigates to the previous step. Disabled on the first form step (Basic Information).
"Submit Inquiry" button: Appears only on the final step. Validates the entire form. If valid, compiles all data and initiates submission to the backend.
Data Persistence (Client-Side): Form data entered in each step must be maintained in the main form state (react-hook-form state) as the user navigates between steps.
Dynamic Field Visibility (Client-Side Conditional Logic): Fields within a step must appear or disappear based on selections in previous fields within the same step or previous steps, as detailed in wedding.text.
Step 1: Basic Information (Triggered after Event Type selection)

Fields to Present (all based on wedding.text "Basic Information" section):
Display Selected Event Type: Show the event type chosen via cards (read-only).
Company Name:
Behavior: Only display this input field if "Corporate" was the selected eventType.
Input: Text.
Billing Address:
Input Group: Street Address, Street Address Line 2 (optional), City, State/Province, Postal/Zip Code.
Contact Name:
Inputs: First Name (required), Last Name (required).
Email:
Input: Email (required, with email format validation).
Phone Number:
Input: Text (with phone number format validation, optional).

Input: Date picker.
Discount Promo Code:
Input: Toggle/Radio (Yes/No).
Conditional Field: "Enter code here" (Text input).
Behavior: Only display if "Discount Promo Code" is "YES".

Step 3: Main Menu Categories & Packages (Dynamic Content)

Requirement: This entire step's content must change based on the requestedTheme selected in Step 2.
Behavior (General Pattern for each theme like "Taco Fiesta", "American BBQ", etc.):
Display Theme Title: e.g., "Taco Fiesta Selections".
Package Selection:
Present available packages for the theme (e.g., Bronze, Silver, Gold, Diamond for Taco Fiesta) with their per-person prices and a summary of inclusions (e.g., "3 proteins, 2 sides...").
Input: Radio buttons or clickable cards.
Conditional Note: If a package has a minimum guest count (e.g., "Bronze package not available for a guest count lower than 50" for Taco Fiesta), and the guestCount from Step 2 is below this minimum, either disable that package option or display a clear warning message.
Item Category Sections (Proteins, Sides, Salsas, etc.):
Behavior: These sections only appear after a package is selected.
For each category relevant to the selected theme and package:
Display category title (e.g., "Proteins").
Display selection limit instructions (e.g., "Please select 3 proteins for the Silver Package"). This limit is determined by the chosen package.
List all available items within that category for the theme (from wedding.text).
Input: Checkboxes for each item.
Upcharges: Clearly display any per-person upcharge next to the item name (e.g., "Flank steak Fajitas - +$2.00 pp").
Selection Limit Enforcement (Client-Side):
As the user checks items, keep a running count of selections for that category.
Once the selection limit for the category (based on the package) is reached, disable all other unchecked item checkboxes in that category.
Provide real-time feedback, e.g., "2 of 3 proteins selected."
Data Storage: Store the chosen package name and lists of selected item names (or unique IDs if you have them) for each category in the form state. Store any selected upcharge items distinctly if needed for calculation.

Input: "Enter number of servings for spreads" (Number input, with quantity options like 24, 36, 48, etc., if these are fixed choices).
