Data Structure for API (Conceptual for Backend Dev):

When saving a rule, you'll likely send an object to the backend looking something like:
JSON

{
  "formId": /* current form's ID */,
  "triggerFormPageQuestionId": /* selectedQuestion.id */,
  "conditionType": "equals",
  "conditionValue": "Yes",
  "actionType": "show_question",
  "targets": [
    { "targetType": "question", "targetId": /* ID of target formPageQuestion */ },
    { "targetType": "page", "targetId": /* ID of target formPage */ }
  ],
  "description": "Show follow-up if user says yes."
}
The backend will then create/update records in formRules and formRuleTargets.
User Experience Considerations:

Clarity: Make the rule builder as intuitive as possible. Use clear labels.
Performance: Fetching allQuestionsInForm might be an issue for very large forms. Consider pagination or searching within the target selection dropdowns if needed.
Preventing Loops: Be cautious about allowing rules that could create infinite loops (e.g., Question A shows Question B, and Question B shows Question A). Simple self-referencing for visibility/requirement is usually disallowed.
Rule Ordering: If multiple rules could apply, their execution order might matter. Your schema has executionOrder in formRules, which is good. The UI doesn't yet manage this, but it's a future consideration.
