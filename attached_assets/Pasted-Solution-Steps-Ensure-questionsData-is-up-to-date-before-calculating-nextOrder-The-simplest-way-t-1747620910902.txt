Ensure questionsData is up-to-date before calculating nextOrder:
The simplest way to address this is to make the displayOrder calculation more robust by fetching the latest count directly or by ensuring the mutation waits for the query to be fresh. However, a more direct fix is to ensure the displayOrder is correctly derived from the current state of the questions on the client, if available, or by fetching count if not.

      title: "No page selected",

  // The queryClient.invalidateQueries should eventually update questionsData.

  // A more robust way to get the current number of questions for the selected page:

Modify Server-Side POST /api/form-builder/pages/:pageId/questions (Optional but good practice):
While the client should send the correct displayOrder, the server could also be made more resilient. Instead of strictly relying on the client-sent displayOrder for new questions, it could calculate the next available displayOrder based on existing questions for that formPageId if the client doesn't provide one or if a conflict occurs. This makes the API more robust.

The core problem is the race condition where the client calculates the next displayOrder based on stale data. Ensuring questionsData.data is accurately reflecting the current list of questions before calculating nextOrder is key. The invalidation mechanism of react-query is supposed to handle this, but rapid sequential additions might execute faster than the refetch and state update cycle. The direct use of questionsData.data as currentQuestionsOnPage at the moment of calculation should be correct if questionsData is indeed updated promptly by react-query after invalidation.
