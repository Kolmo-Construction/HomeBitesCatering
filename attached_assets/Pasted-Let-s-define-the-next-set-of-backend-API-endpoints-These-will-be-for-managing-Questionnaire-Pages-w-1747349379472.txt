Let's define the next set of backend API endpoints. These will be for managing Questionnaire Pages within a specific QuestionnaireDefinition.

Base Path for these Page-related Endpoints:
All routes will be prefixed, for example, under /api/admin/questionnaires/definitions/:definitionId/pages.
The :definitionId path parameter will be crucial to ensure pages are managed within the context of their parent questionnaire definition.

Here are the suggested endpoints, their details, and function names you might use for the Express handlers:

1. Create a New Page for a Definition

HTTP Method & Route: POST /api/admin/questionnaires/definitions/:definitionId/pages
Handler Name Suggestion: handleCreateQuestionnairePage(req, res)
Path Parameters:
definitionId: (integer) The ID of the QuestionnaireDefinition this page belongs to.
Request Body (JSON):
title: (string, required, e.g., min 3 characters) The title of the page (e.g., "Contact Information", "Event Details").
order: (integer, required, non-negative) The numerical order of this page within the definition.
Zod Schema Suggestion for Request Body: QuestionnairePageCreateSchema
Response (Success - 201 Created):
The newly created page object (matching the structure of questionnairePages table record).
Key Logic:
Validate definitionId (must be a positive integer).
Verify that the QuestionnaireDefinition with the given definitionId exists.
Validate the request body using the Zod schema.
Insert a new record into the questionnairePages table, ensuring definitionId is correctly set.
The createdAt and updatedAt fields should be set automatically by the database or your ORM.
2. List All Pages for a Definition

HTTP Method & Route: GET /api/admin/questionnaires/definitions/:definitionId/pages
Handler Name Suggestion: handleListQuestionnairePages(req, res)
Path Parameters:
definitionId: (integer) The ID of the QuestionnaireDefinition.
Request Body: None
Response (Success - 200 OK):
An array of page objects belonging to the specified definitionId.
The array should be ordered by the order field (ascending).
Key Logic:
Validate definitionId.
Verify that the QuestionnaireDefinition with the given definitionId exists.
Query the questionnairePages table, filtering by definitionId.
Sort the results by the order column.
3. Get a Single Page Details

HTTP Method & Route: GET /api/admin/questionnaires/definitions/:definitionId/pages/:pageId
Handler Name Suggestion: handleGetQuestionnairePage(req, res)
Path Parameters:
definitionId: (integer) The ID of the QuestionnaireDefinition.
pageId: (integer) The ID of the specific QuestionnairePage.
Request Body: None
Response (Success - 200 OK):
A single page object.
Response (Error - 404 Not Found): If the page or definition doesn't exist, or if the page doesn't belong to the specified definition.
Key Logic:
Validate definitionId and pageId.
Query the questionnairePages table for the record matching pageId.
Verify that the fetched page's definitionId matches the definitionId from the path parameter.
4. Update a Page

HTTP Method & Route: PUT /api/admin/questionnaires/definitions/:definitionId/pages/:pageId
Handler Name Suggestion: handleUpdateQuestionnairePage(req, res)
Path Parameters:
definitionId: (integer) The ID of the QuestionnaireDefinition.
pageId: (integer) The ID of the page to update.
Request Body (JSON):
title: (string, optional, e.g., min 3 characters) The new title for the page.
order: (integer, optional, non-negative) The new order for the page.
Zod Schema Suggestion for Request Body: QuestionnairePageUpdateSchema
Response (Success - 200 OK):
The updated page object.
Response (Error - 404 Not Found): If the page or definition doesn't exist, or if the page doesn't belong to the specified definition.
Key Logic:
Validate definitionId and pageId.
Validate the request body.
Fetch the page to ensure it exists and belongs to the correct definitionId.
Update the specified fields in the questionnairePages table for the given pageId.
Update the updatedAt timestamp.
Note on order changes: If order is updated, you might need to consider if this requires re-ordering other sibling pages. For a simple PUT, you could just update the value. A more complex reordering is often handled by a dedicated batch endpoint (see optional endpoint below).
5. Delete a Page

HTTP Method & Route: DELETE /api/admin/questionnaires/definitions/:definitionId/pages/:pageId
Handler Name Suggestion: handleDeleteQuestionnairePage(req, res)
Path Parameters:
definitionId: (integer) The ID of the QuestionnaireDefinition.
pageId: (integer) The ID of the page to delete.
Request Body: None
Response (Success - 200 OK or 204 No Content):
A success message (e.g., { "message": "Page deleted successfully" }) or the deleted page object.
Response (Error - 404 Not Found): If the page or definition doesn't exist, or if the page doesn't belong to the specified definition.
Key Logic:
Validate definitionId and pageId.
Verify that the page exists and belongs to the specified definitionId.
Delete the record from the questionnairePages table.
Important Consideration: Due to the onDelete: 'cascade' foreign key constraint from questionnaireQuestions.pageId to questionnairePages.id (as defined in the schema earlier), deleting a page will automatically delete all questions associated with that page, and subsequently, all options/matrix columns associated with those questions. Ensure this cascading behavior is intended and handled gracefully.
6. (Optional but Recommended) Batch Update Page Order for a Definition

HTTP Method & Route: PATCH /api/admin/questionnaires/definitions/:definitionId/pages/reorder
Handler Name Suggestion: handleReorderQuestionnairePages(req, res)
Path Parameters:
definitionId: (integer) The ID of the QuestionnaireDefinition whose pages are being reordered.
Request Body (JSON):
pageIds: (array of integers, required) An array of pageIds in the new desired order. (e.g., [3, 1, 2])
Zod Schema Suggestion for Request Body: QuestionnairePagesReorderSchema
Response (Success - 200 OK):
An array of the updated page objects in their new order, or a success message.
Key Logic:
Validate definitionId.
Validate the pageIds array in the request body.
Verify that all pageIds in the array belong to the specified definitionId and that the set of IDs matches all existing pages for that definition (to avoid errors or orphaned pages).
Iterate through the provided pageIds array. For each pageId, update its order field to reflect its new position (index in the array). This should ideally be done within a database transaction to ensure atomicity.
Update the updatedAt timestamp for all affected pages.
General Considerations for All Endpoints:

Authentication & Authorization: Ensure all these routes are protected by your admin authentication and authorization middleware.
Error Handling: Implement robust error handling (e.g., 400 for bad requests/validation errors, 401/403 for auth issues, 404 for not found, 500 for server errors).
Database Interaction: Use your Drizzle ORM (db instance) and the schemas from shared/schema.ts for all database operations.
Transaction Management: For operations that involve multiple database writes (like the reorder endpoint, or if creating a page involved updating a count on the definition), use database transactions to ensure data consistency.
You can now proceed to implement these route handlers in your server/routes.ts file, likely by extending the adminQuestionnaireRouter you've already started or by creating a new router that's then mounted onto it.
