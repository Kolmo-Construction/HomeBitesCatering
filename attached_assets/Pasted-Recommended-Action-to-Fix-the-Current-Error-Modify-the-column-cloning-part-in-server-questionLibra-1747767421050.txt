Explanation for the Fallback:

column.cell_input_type || 'text': If column.cell_input_type is null, undefined, or an empty string, it will use 'text' as the value for the new cloned column. This ensures the NOT NULL constraint is met.
column.column_order || 0: Similarly, providing a fallback for column_order (and row_order in the rows loop) ensures that if these are unexpectedly null, the database default or a 0 is used, preventing potential issues. Your schema already has default(0) for these, so column.column_order being null should be handled by the DB, but explicit fallback is safer.
