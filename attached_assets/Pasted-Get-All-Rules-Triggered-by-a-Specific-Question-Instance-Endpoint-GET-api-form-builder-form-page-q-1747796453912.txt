Get All Rules Triggered by a Specific Question Instance

Endpoint: GET /api/form-builder/form-page-questions/:questionInstanceId/rules
Purpose: Retrieves all rules where the specified questionInstanceId (this is formPageQuestions.id) is the trigger. Used by the FormEditor's "Logic" tab.
Logic:
Parse and validate questionInstanceId from req.params.
Fetch all rules from formSchema.formRules where triggerFormPageQuestionId equals questionInstanceId.
For each rule, fetch its targets from formSchema.formRuleTargets.
Response:
Success (200): An array of rule objects with their targets (same structure as endpoint #2).
Error (404/500): Error message.
4. Update an Existing Rule

Endpoint: PUT /api/form-builder/rules/:ruleId
Purpose: Updates the details of an existing conditional rule, including its targets.
Request Body Schema: Same as the create endpoint, but all fields might be optional for partial updates (though typically, a full update is easier to manage for rules).
Logic:
Parse and validate ruleId from req.params.
Parse and validate the request body.
Use db.transaction(async (tx) => { ... }).
Update the specified rule in the formSchema.formRules table.
Delete all existing target records from formSchema.formRuleTargets associated with this ruleId.
Iterate through the new targets array from the request body and insert them into formSchema.formRuleTargets.
Fetch the updated rule with its new targets to return.
Response:
Success (200): The updated rule object with its targets.
Error (400/404/500): Error message.
5. Delete a Rule

Endpoint: DELETE /api/form-builder/rules/:ruleId
Purpose: Deletes a conditional rule and its associated targets.
Logic:
Parse and validate ruleId from req.params.
Delete the rule from formSchema.formRules where id matches ruleId.
Ensure your database schema for formRuleTargets has onDelete: 'cascade' for the ruleId foreign key. If so, associated targets will be deleted automatically.
If not using cascading deletes at the DB level, you must explicitly delete records from formSchema.formRuleTargets where ruleId matches before deleting the rule itself.
Response:
Success (200 or 204): Success message (e.g., { message: "Rule deleted successfully" }).
Error (404/500): Error message.
Implementation Notes for server/formBuilderRoutes.ts:

Zod for Validation: Strongly recommend using Zod to validate request bodies and parameters for each endpoint to ensure data integrity.
Error Handling: Implement robust error handling for database operations and invalid inputs, returning appropriate HTTP status codes and error messages.
Drizzle ORM Queries:
For creating and updating rules with targets, transactions are crucial.
When fetching rules with targets, you might perform two queries (one for rules, then loop and query for targets) or look into Drizzle's capabilities for joining and shaping related data in a single query if possible (though relations are primarily for query building, direct shaping might need manual assembly).
