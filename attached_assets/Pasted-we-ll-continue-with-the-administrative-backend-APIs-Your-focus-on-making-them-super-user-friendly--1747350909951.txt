While much of the user experience will be shaped by the frontend interface, the backend API needs to be robust, logical, and allow for efficient management of all form components.

The next set of administrative endpoints will be for managing Questions within a specific QuestionnairePage. This means defining how administrators can add, view, update, and remove various types of questions (text fields, dropdowns, matrices, etc.) on each page of a questionnaire.

Here's the plan for these Question-related API endpoints:

Base Path for Question-related Endpoints:

We'll primarily operate on questions in the context of the page they belong to.
Example: /api/admin/questionnaires/pages/:pageId/questions
The :pageId path parameter is essential. We'll also need endpoints to get/update/delete individual questions by their unique questionId.

1. Create a New Question on a Page

HTTP Method & Route: POST /api/admin/questionnaires/pages/:pageId/questions
Handler Name Suggestion: handleCreateQuestionnaireQuestion(req, res)
Path Parameters:
pageId: (integer) The ID of the QuestionnairePage this question belongs to.
Request Body (JSON): This will be a comprehensive payload.
questionText: (string, required) The actual text of the question (e.g., "What is your event date?").
questionKey: (string, required, globally unique) A unique machine-readable key for this question (e.g., event_date, guest_count_estimate). This is critical for data mapping and conditional logic.
questionType: (enum string, required, from questionTypeEnum in your schema: e.g., 'text', 'select', 'matrix_single').
order: (integer, required) The numerical order of this question within the page.
isRequired: (boolean, optional, default: false).
placeholderText: (string, optional) Placeholder text for input fields.
helpText: (string, optional) Additional explanatory text displayed with the question.
validationRules: (JSON object, optional) Rules for validation (e.g., {"minLength": 5, "pattern": "^\\d+$", "min": 0, "max": 100}). Structure depends on questionType.
options: (array of objects, optional, required for types like 'select', 'radio', 'checkbox_group')
Each object: {"optionText": "string", "optionValue": "string", "order": "integer", "defaultSelectionIndicator": "string (optional)", "relatedMenuItemId": "integer (optional)"}
matrixColumns: (array of objects, optional, required for types like 'matrix_single', 'matrix_multi')
Each object: {"columnText": "string", "columnValue": "string", "order": "integer"}
Note on Matrix Rows: For matrix questions, the row labels can often be managed using the options array, where each "option" represents a row.
Zod Schema Suggestion: QuestionnaireQuestionCreateSchema. This schema will need to be quite sophisticated, potentially using Zod's discriminatedUnion based on questionType to validate the presence and structure of options or matrixColumns.
Response (Success - 201 Created):
The newly created question object, including its generated id, and any nested options or matrixColumns that were created and their IDs.
Key Logic:
Validate pageId and ensure the parent page exists.
Crucially, validate questionKey for global uniqueness across the questionnaireQuestions table.
Validate the request body.
In a database transaction:
Insert into questionnaireQuestions.
If options are provided (and valid for the questionType), insert them into questionnaireQuestionOptions, linking to the new question's ID.
If matrixColumns are provided (and valid for the questionType), insert them into questionnaireMatrixColumns, linking to the new question's ID.
2. List All Questions for a Page

HTTP Method & Route: GET /api/admin/questionnaires/pages/:pageId/questions
Handler Name Suggestion: handleListQuestionnaireQuestions(req, res)
Path Parameters:
pageId: (integer) The ID of the QuestionnairePage.
Response (Success - 200 OK):
An array of question objects belonging to the page, ordered by their order field.
Each question object should also include its associated options and matrixColumns (if any, as nested arrays).
Key Logic:
Validate pageId.
Fetch all questions for the pageId.
For each question, eagerly load or subsequently fetch its related options and matrixColumns. Drizzle's relational queries are ideal here.
3. Get a Single Question's Details

HTTP Method & Route: GET /api/admin/questionnaires/questions/:questionId
(Using questionId as it's globally unique and the primary key for the question itself.)
Handler Name Suggestion: handleGetQuestionnaireQuestion(req, res)
Path Parameters:
questionId: (integer) The ID of the question.
Response (Success - 200 OK):
A single question object, including its options and matrixColumns (if any).
Key Logic:
Validate questionId.
Fetch the question and its related options and matrixColumns.
4. Update a Question

HTTP Method & Route: PUT /api/admin/questionnaires/questions/:questionId
Handler Name Suggestion: handleUpdateQuestionnaireQuestion(req, res)
Path Parameters:
questionId: (integer) The ID of the question to update.
Request Body (JSON):
Similar to the creation payload, but all fields are optional.
Allows updating questionText, questionKey (ensure uniqueness if changed), questionType, order, isRequired, placeholderText, helpText, validationRules.
Updating options or matrixColumns: The most straightforward approach for user-friendliness in the API is to expect the entire new set of options/columns. The backend would then:
Delete all existing options/columns for this questionId.
Insert the new ones provided in the request. This strategy simplifies frontend logic significantly compared to trying to patch individual sub-items.
Zod Schema Suggestion: QuestionnaireQuestionUpdateSchema.
Response (Success - 200 OK):
The updated question object, including its (potentially new) options or matrixColumns.
Key Logic:
Validate questionId and ensure the question exists.
Validate the request body. If questionKey is being changed, re-validate its uniqueness.
Changing questionType: This is a complex part. If questionType changes (e.g., from 'text' to 'select'), the requirements for options/matrixColumns change. The request body should reflect the new type's needs. Existing options/columns for the old type might need to be cleared.
Use a database transaction for all updates, especially when modifying the question and its nested options/columns.
5. Delete a Question

HTTP Method & Route: DELETE /api/admin/questionnaires/questions/:questionId
Handler Name Suggestion: handleDeleteQuestionnaireQuestion(req, res)
Path Parameters:
questionId: (integer) The ID of the question to delete.
Response (Success - 200 OK or 204 No Content):
A success message.
Key Logic:
Validate questionId.
Delete the question from questionnaireQuestions.
The onDelete: 'cascade' in your Drizzle schema for questionnaireQuestionOptions.questionId and questionnaireMatrixColumns.questionId should automatically delete associated options and columns.
Important Consideration: Deleting a question that is used in questionnaireConditionalLogic (via its questionKey) will orphan those logic rules on the DB level since the keys are text. Your application might need a way to identify or clean up such orphaned rules, or the UI should warn the admin.
6. (Optional but Recommended) Batch Update Question Order on a Page

HTTP Method & Route: PATCH /api/admin/questionnaires/pages/:pageId/questions/reorder
Handler Name Suggestion: handleReorderQuestionnaireQuestions(req, res)
Path Parameters:
pageId: (integer) The ID of the QuestionnairePage.
Request Body (JSON):
questionIds: (array of integers, required) An array of questionIds in the new desired order.
Zod Schema Suggestion: QuestionnaireQuestionsReorderSchema.
Response (Success - 200 OK):
An array of the updated question objects in their new order, or a success message.
Key Logic:
Similar to reordering pages: validate pageId, validate questionIds array (ensure all IDs belong to the page and are comprehensive).
Update the order field for each question in a transaction.
User-Friendliness Notes for these APIs:

Comprehensive Payloads: Allowing full creation/update of a question including its options/columns in a single API call makes the frontend development easier.
Clear questionKey: Emphasize the importance and uniqueness of questionKey for reliable data handling and logic.
Cascade Deletes: Leveraging database cascade deletes for options/columns simplifies backend logic.
Ordering: Providing explicit ordering fields and reordering endpoints supports drag-and-drop interfaces on the frontend.
